## Project Room purpose

A project room is an evidence dossier, not a generic portfolio page.

It must:
- prove;
- explain;
- show runtime evidence;
- show architecture honestly;
- connect the project to capabilities;
- route to the live product;
- route to related systems.

## Orbe purpose

Orbe is the public intelligence layer of the lab.

It should:
- guide;
- remember;
- connect context;
- help explore the portfolio;
- help shape a project;
- help build brand foundations;
- persist important decisions;
- continue sessions without forced login.

Orbe is not merely a chatbot.

## Project / Branding reality

Project creation and branding must converge into one shared persistent workspace.

There must not be independent copies of:
- palette logic;
- logo flow;
- reference image flow;
- storyboard flow;
- asset persistence;
- project-card identity.

## Known gaps to verify

- mixed-language UI surfaces;
- generic Project Room welcome copy;
- Guided Tour too shallow and possibly spending unnecessary tokens;
- project and branding flows connected but not fully unified;
- generic one-off visual generation;
- oversized generated images in chat;
- no coherent asset vault/workbench;
- admin sessions centered on transcript instead of dossier;
- project-room evidence sometimes atmospheric instead of factual;
- stale/contradictory documentation;
- heartbeat scheduling must be verified in deployment;
- request-time translation latency must be evaluated;
- multilingual guardrail coverage must be tested;
- production network exposure must be audited.

## Stability

Preserve where practical:
- public URLs;
- project identifiers;
- tier/category IDs;
- storage-driver abstractions;
- working tests;
- `/os`;
- current Hall/Featured/Archive data semantics.

Public naming can change without renaming persisted IDs.
