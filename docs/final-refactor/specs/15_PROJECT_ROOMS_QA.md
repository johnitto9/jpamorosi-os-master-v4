# 15_PROJECT_ROOMS_QA

## Objective

Raise project rooms from beautiful dossiers to credible evidence systems.

## Runtime proof

A section titled like `See the system running` must show actual project behavior.

Acceptable:
- product recording;
- real flow;
- mobile/web interaction;
- order lifecycle;
- merchant/admin behavior.

Not acceptable as sole proof:
- decorative particles;
- unrelated ambient loop.

## Architecture honesty

Separate:

### System topology
- Web/PWA
- Mobile wrapper
- API
- Data
- Payments
- Messaging
- Storage

### Product flow
- Browse
- Reserve
- Checkout
- Pickup
- Settle

Do not call a purely functional flow "architecture" without context.

## Evidence Wall

Evidence should be heterogeneous.

Mix:
- product UI;
- operation;
- store listing;
- real feedback;
- order flow;
- admin;
- campaign assets.

Do not make Evidence equal marketing gallery.

## Proof Cards

Avoid four variations of "it is real".

Use distinct dimensions:
- Production
- Product
- Engineering
- Operations

## Founder Note

Do not repeat hero copy.

Reveal:
- product learning;
- constraint;
- insight;
- real friction.

## Stack truth

Every visible badge maps to current project truth.

No:
- aspirational tools;
- planned tools;
- cross-project tools;
- stale historical tools.

## Vertical rhythm

Project rooms should be denser than Home.

Audit:
- dead gaps;
- blurred preview sections;
- excessive scroll tax.

## Animation safety

Test:
- fast scroll;
- refresh mid-page;
- direct deep link;
- reduced motion.

No content may remain stuck blurred/transparent.

## Acceptance

- runtime proof real;
- architecture reflects actual system;
- evidence wall diverse;
- founder note unique;
- stack verified;
- reduced-motion complete.
